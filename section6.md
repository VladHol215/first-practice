## Розділ 6. Архітектура системи та залежності

### 6.1. Тип архітектури
Система побудована на основі **трирівневої архітектури (3-tier architecture)**. Це дозволяє розділити відповідальність між компонентами, полегшує тестування та підвищує загальний рівень безпеки фінансових даних.

### 6.2. Детальний опис рівнів:

1.  **Presentation Layer (Рівень представлення):**
    * **Front-end:** Реалізований як веб-додаток (React/Angular) та мобільний клієнт (iOS/Android).
    * **Функція:** Відображення інтерфейсу, збір даних від користувача та візуалізація стану рахунків. Взаємодіє з сервером через захищені RESTful API запити.

2.  **Application Layer (Рівень бізнес-логіки):**
    * **Back-end сервер:** Обробляє основні запити. Тут реалізовані сервіси перевірки балансу, валідації лімітів та розрахунку комісій.
    * **Забезпечення безпеки:** На цьому рівні відбувається перевірка токенів авторизації (JWT) та шифрування транзакцій перед записом.

3.  **Data Layer (Рівень даних):**
    * **База даних:** Реляційна БД (наприклад, PostgreSQL або MySQL) для збереження структурованих даних про клієнтів та історію платежів.
    * **Кешування:** Використання Redis для прискорення доступу до часто запитуваних даних (наприклад, поточні курси валют).

### 6.3. Залежності між підсистемами
Система складається з кількох модулів, що мають чіткі зв'язки:
* **Модуль аутентифікації:** Є вхідною точкою. Усі інші підсистеми залежать від нього, оскільки вимагають підтвердженого ID користувача.
* **Модуль обробки транзакцій:** Має зовнішню залежність від **банківського процесингового центру** через захищений шлюз. Без відповіді від банку транзакція не може бути фіналізована.
* **Сервіс сповіщень:** Залежить від модуля транзакцій. Після успішної операції він ініціює відправку SMS або Push-повідомлення через зовнішні шлюзи (наприклад, Firebase або Twilio).

### 6.4. Висновок по архітектурі
Обрана модульна структура дозволяє оновлювати окремі частини системи (наприклад, змінити дизайн фронтенду) без зупинки роботи всього бекенду або бази даних.
